Codex 教程 – 零基础 VibeCoding 全栈开发保姆级指南

AI教程2个月前更新 AI小集
0 6

近期 Codex 的动作很密集，方向也很明显。Codex 不再只是“帮程序员写代码”，OpenAI 正在把 Codex 推成一个能长期干活、能接企业流程、能进安全场景的 Agent 平台。

OpenAI 这一波，就是要给Claude Code松松土了。

我们先来看看Codex、Claude Code 和 OpenClaw 的区别，以及什么人更适合Codex。

作为一个0代码基础的小白，这几个产品我都用过，个人感觉Codex会更适合我。

 

01. 怎么用？很简单

 

进入 ChatGPT 首页，左侧导航栏有一个 Codex 。

ChatGPT 首页：https://chatgpt.com/

点击Codex，进入下载页面，它目前有CLI、App和IDE三种使用方式。我用的是App的方式。下载也很快，我大概2分钟，就下载好了。

下载完之后，和你安装其他软件一样，把图标拖动到Applications中，然后一步一步按照提示走。安装好之后点击图标。首次登陆时，会让你选择登录方式。

订阅了ChatGPT Plus/Pro会员的，选择“使用ChatGPT继续”。会跳转到网页走一次授权，输入登录账号就行，整个过程很快，大概花了10秒。登录好之后，Codex用的就是订阅套餐里的GPT-5.5额度，不需要另外付费了。

没有订阅的友友，可以去 platform.openai.com 生成一个 API Key，然后粘贴进 App，按 token 计费用。不过不推荐这种方式，万一用超了，肉疼。建议先用第一种方式感受一下Codex的消耗情况再说。

登陆之后，就可以看到首页对话框长这样。

Codex真正牛的是它预置了一大批技能和插件。

把Codex最核心的插件和Skills做了个汇总。

Codex了解得差不多了，我们直接开始实测。

 

02. 全栈开发

 

我这次要用Codex做一个媒介投放的KOL派单平台。

这个 case 足够大，也足够贴近真实业务。它能同时验证Codex 产品理解、前端生成、业务流程结构化、交易链路设计、Browser Use 自测、云服务器部署上线这几个能力。

这个case真正难的地方不在于页面好不好看，而在页面里能不能帮我判断：这个达人适合什么 AI 产品、凭什么说他适合、内容证据在哪里、报价是否确认、履约风险在哪里，以及客户能不能直接选择套餐、填 brief、提交下单？

直接把提示词甩给Codex，Codex 生成的页面，整体方向是对的。

前端页面搞定了，但是感觉太丑了，我直接让Codex参考苹果官网帮我重新设计前端。这次展示效果就好多了。

接着完成后端，再买个云服务器部署上线。代码小白也不要紧，直接让Codex一步一步指导就行。

在Codex的指导下，在终端一步一步跑起来。

如果遇到报错，直接复制代码让Codex查看即可。根据反馈，再继续进行下一步操作。这都不难，就是需要有耐心，一点一点磨。

购买云服务按照这个配置就够用了。

然后是安装服务器Docker，以及上传项目到云服务器。

Codex还可以自动排查安全隐患，帮我们进行修复。

然后我这个名叫KOL UP的媒介投放网站就上线了。

这个项目开发流程是这样的，整个过程自动化程度非常高，我只需要描述需求，然后Codex干活，我回来验收是否实现就可以。

如果打分，我会给 Codex 95 分，还有5分扣在响应速度上。

 

03. Codex的10大核心功能

 

在全栈开发场景Codex足以胜任大部分任务，但Codex不仅仅是一个开发工具，接下来我们介绍一下Codex的10大核心功能，这一波升级真的又强又贴心。

本地文件操作

Codex 真正有用的前提，是它能直接进入你的项目文件里工作。

它能看目录、改文件、查配置、修页面、调整样式，本地文件操作，这也是 Codex 从“问答工具”变成“干活工具”的分界线。

上下文管理

一个项目不是靠一句提示词完成的。所以当你的上下文太长的时候，Codex就会这样，别问我是怎么知道的。

Codex 虽然可以帮你压缩旧对话，但压缩也会耗时，而且压缩后的上下文不一定保留你真正关心的细节。

用 Codex的时候，一定要把任务拆成短会话。一个会话只做一件事，规划、实现和报错修复，放在不同的会话里。不要在一个会话里连续做“设计页面、改前端、改后端、部署、修 bug、再改视觉”。这种最容易把上下文撑爆。

你还可以让 Codex 在关键节点帮你写一个项目交接文件。

提示词：请把当前项目状态、已完成内容、关键文件、下一步任务、注意事项，整理到 docs/CODEX_HANDOFF.md。内容要给下一个 Codex 会话直接接手使用。

之后每次新开会话，只要说：

提示词：请先阅读 docs/CODEX_HANDOFF.md，然后继续执行下一步。

命令行工具使用

很多非程序员真正卡住的地方，是不会装环境、跑命令、处理报错。

比如安装软件、配置 CLI、安装 MCP、安装 Skill、运行项目、卸载依赖、检查版本，这些事情对程序员很普通，对文科生很劝退。

现在Codex 能帮你处理命令行，就相当于帮你跨过了技术门槛里最吓人的一关，而且还非常简单，只需要在对话框中直接提需求就行。

预览浏览器及注释功能

这个功能对非程序员特别友好。因为你不需要看代码判断结果，只需要看页面效果。Codex 写完网页之后，你可以直接在预览浏览器里看按钮有没有错位、页面好不好看、交互有没有问题。

还能通过注释功能，针对性地进行修改。

流程沉淀为Skills

我们日常的工作流，如果每次都重新讲需求，会非常浪费。把流程沉淀成 Skill，就相当于把你的审美、模板、经验、标准变成可复用资产。

当然，你可以可以直接安装别人的Skills。

持久记忆

持久记忆解决的是“不要每次从头解释”的问题。全局记忆更像你的长期偏好，比如你喜欢什么写作风格、封面风格、代码风格、常用工具。可以在【设置】-【个性化】里面进行配置。

实验记忆也在【设置】-【个性化】中。

并行任务处理

这个能力非常关键了，比如你同时让 Codex 做三件事：一个任务修网页 Bug，一个任务优化页面视觉，一个任务整理 README。这样效率会高很多。

对初学者来说，一开始并行太多，容易乱。对进阶用户来说，并行任务非常有用，可以让 Codex 像一个小团队一样同时推进多个方向。

模型选择和额度查询

你需要知道什么时候用强模型，什么时候用轻量模型。

简单任务，比如改文案、调按钮、解释文件，可以用便宜或轻量的模型。复杂任务，比如重构项目、修深层 Bug、做完整网页，就需要更强模型。这个能力的价值在于帮你节省额度、提高成功率。

另外，还能在这里能看到上下文使用占比，用起来就很安心。

还能看到整个会员额度使用情况，有一说一，Codex的额度还是给的很大方的。

图片生成

图片生成直接调用Image2的能力，非常强大。Codex 更偏项目实现，图片生成更偏内容包装。

你可以把它们组合起来，Codex 做网页、工具、互动页面，图片生成做封面、插图、视觉物料。

插件

插件的价值取决于生态成熟度和你的具体需求。如果插件能帮你接入 GitHub、数据库、浏览器、设计工具、自动化工具，那就很有价值。

但插件本身不是越多越好。对非程序员来说，插件太多反而容易分散注意力。

你真正需要关注的是，这个插件能不能帮你完成一个明确流程，比如发布网页、读取文件、连接素材库、生成报告。

 

04. 一些分享

 

这次实测和体验之后，我的体感是Codex 在写代码方面未必见得比Claude Code牛，但是它在不封号，不搞七七八八限制情况下，能实现这个难度的需求，已经很抗打了。

Claude Code 在复杂代码仓库、终端开发、重构和工程流程里依然很强。

但 Codex 的优势在于，它把 Browser Use、Computer Use、Image 2 、Skills、插件放在一个工作流里。比如做一个前端页面，Codex 不只是写代码，还能打开页面看效果，继续改布局，后面还能生成配图、图标、产品展示素材。对非程序员来说，这种“能看见结果、能继续迭代”的体验更直观。

Codex 负责把代码写出来，人负责把业务想明白。真正的差距，藏在真实需求洞察和流程拆解里。
